## Introduction

Forming a platform that is able to operate is just a segment of the entire work. Another part is to ensure that the system can work well for everyone and can be fully evaluated when done. This post will showcase our testing methods for the Critique Canvas and the functions we implemented into the platform.

## Why Accessibility Matters for Critique Canvas

Critique Canvas is a digital art application for the community; therefore, it aims to be open to every artist. If the annotation feature is only operated by a mouse, then the audience that uses a keyboard or a touchscreen cannot go through the whole experience. Our team's viewpoint is that accessibility is one of the core factors for our project, not an option that we consider.

## The Accessible Fallback List

While annotation pins on the image are visual attributes, a screen reader will struggle to locate their positions. We tackle this issue by implementing an accessible fallback: a written list of all annotations displayed below the image. The list will showcase identical data to the pin, which includes comments, categories and the author. Non-mouse users can still read the annotation without the need to interact with the canvas.

## Keyboard Navigation and Form Labels

In our platform, all input forms come with a label. The annotation form inside the detail page applies aria-label attributes to the text area, as well as choosing attributes to enable the screen reader to show the purpose of each field. Filter pills will be seen as button elements instead of styled divs, so they can be recognisable from the keyboard.

## How We Will Evaluate the App

Evaluating the app means checking three things. First, does it work — can a user upload an artwork, leave an annotation, and filter by category without errors? Second, is it accessible, where the same tasks can be completed using only a keyboard? Third, does the interface make it clear what to do at each step? We plan to test these by manually analysing each user flow, and identifying that all interactive attributes can be accessible and have the correct label on them.

## What We Still Need to Improve

The filter pills currently update the sidebar annotation list via HTMX, but do not update the pins on the canvas at the same time. This means the visual and accessible versions of the annotations can get out of sync. Fixing this is our next priority because it affects both the usability and the accessibility of the filter feature.

## Data Privacy and Responsibility

Another crucial factor is user data management. Critique Canvas saves the names of users and their comments publicly; therefore, each individual in the audience can view the commentator and their feedback. We made this design decision intentionally because our concept values transparency in feedback. This also means users need to understand that their name will be visible. The platform does not collect personal information, therefore the risk of privacy issues will be low. One of the potential future implementations is the ability for users to delete their comments, allowing them control over their public messages.
