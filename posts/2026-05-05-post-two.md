## From Concept to Code

By identifying the core functions of the Critique Canvas, the next stage is to identify whether our team can develop this platform based on our technical skills. This post will identify whether those main features are achievable and the technical decisions that were developed from them.

## Is the Annotation Feature Feasible?

While the click-to-annotate part looks straightforward, it consists of multiple moving features. When a user interacts with an image, the app needs to lock in the precise location of the click, change it to a coordinate pair, send it to the server along with a comment, save it to the database, and then display a pin back on the image without requiring a page refresh.

Each step is manageable. The hard part is how to make them collaborate in a reliable way. After prototyping the interaction, I concluded that this is achievable within our system, but only if each piece is handled carefully. The coordinate capture and storage, in particular, needs a specific approach.

## The Coordinate Problem

If we store annotation positions as pixel values, a pin saved at position 340px, 220px on a 1200px-wide display will appear in completely the wrong place on a 600px-wide screen. This is a crucial problem for any platform that needs to work on multiple devices.

Our team decided to store the coordinates as a percentage of the image dimensions instead of the exact values of the pixels. When a user interacts with the image, the width and height percentages of the image will be calculated, and those values will be stored. CSS will then change the percentage back into pixel coordinates that can correlate with the size of the image when rendering pins. This solution enables the pin to stay exact on multiple different screen sizes without the need of caculation.

## Why HTMX Over a Full JavaScript Framework

One of our team's most crucial core decisions is choosing HTMX instead of React or Vue for developing page updates. The reason for this decision is that we want our page to be able to respond to user interaction in a reliable way that does not require the need for our page to reload. For example, when a user pins a comment, we want that comment to immediately show on the image, without the need for the user to reload, which will result in inconvenience and unreliability. HTMX can solve this problem without needing to build a new layer code.

## Learning experience from fixing the Backend

When a user uploads an image, the code must be able to locate the file to move it to the gallery. I had an issue when building the backend. The system was saving the file in the full address of the location, which includes the city, country and postcode. In reality, the code was only looking for the short version of the location. This results in the image being lost, which leads to a broken link. The image was able to appear when I fixed this issue.

## Next steps

The next step of our work is to fix the "Composition" section. Currently, this part is unable to filter and separate the types of comments that can be pinned to the image. At the same time, we have to make sure that this element can work collaboratively with the pins on the image. For example, when we filter the comments on colours, we expect the filtered comments to appear in the "Composition" section as well as in the image.
